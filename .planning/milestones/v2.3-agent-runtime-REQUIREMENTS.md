# Requirements: Memroos v2.3 — Agent Runtime Enhancements

*Created: 2026-05-10*

## Goal

Hardening of the Hermes agent runtime layer (middleware, memory client, observability) to complement v2.1 security/trust infrastructure and v2.2 LLM optimization. These specs operate at the **agent-runtime layer** (Hermes side) while consuming data produced by Memroos-side security, audit, and model telemetry systems.

## Reconciliation with v2.1/v2.2

The three specs below are **separate from v2.1 Phases 42-45** (security/trust) and **v2.2 Phases 46-49** (LLM optimization/evaluation). They target the **Hermes agent runtime** rather than the Memroos server. Integration points are noted per phase.

| My Spec | v2.1 Phase Relationship | Verdict |
|---------|------------------------|---------|
| Agent Middleware (Phase 50) | Feeds audit data into v2.1 security events and model outcome data into v2.2 telemetry | Complementary, different layer |
| Agent Memory v2 (Phase 51) | Uses Phase 37 memory API and v2.2 recommendation workspace patterns | Client-side enhancement |
| Agent Observability Dashboard (Phase 52) | Consumes v2.1 security events, v2.2 model telemetry, and Phase 50 outcome logs | UI layer on top of runtime data |

**These are NOT overlapping with v2.1 or v2.2 phases.** They are a proposed **v2.3** milestone. Do not reuse phase numbers 42-49.

## Requirements

### Phase 50 — Agent-Side Middleware

**MWMID-01** — Middleware registry: `~/.hermes/middleware/pre/` and `~/.hermes/middleware/post/` directories; each `.py` exports `on_pre_call(ctx)` and/or `on_post_call(ctx)`; loaded at agent startup in alphabetical order.
- Current: No middleware exists — tools execute directly
- Target: Middleware chain executes around every tool call (native, MCP, shell, browser)
- Acceptance: Registering 2 pre + 2 post middleware files; making a tool call triggers all 4 hooks in order; each hook logs to `~/.hermes/logs/middleware.log`

**MWMID-02** — Input validation hook: `pre/01-validate-input.py` validates tool inputs against schemas, rejects with `ValidationError` before tool execution.
- Acceptance: Missing required field returns structured validation error without executing tool; valid inputs pass through unchanged

**MWMID-03** — Secret redaction hook: `pre/02-redact-secrets.py` scans inputs against `~/.hermes/middleware/secret-patterns.json`; replaces matched values with `[REDACTED]` in logs only (tool receives original value).
- Acceptance: Tool call with secret-pattern value has it replaced with `[REDACTED]` in logged input; tool receives original value unchanged

**MWMID-04** — Outcome logging hook: `post/01-log-outcome.py` records tool name, input hash (SHA-256), success/failure, error type, duration_ms, timestamp to `~/.hermes/logs/tool-outcomes.jsonl`.
- Acceptance: 5 mixed tool calls produce 5 JSONL lines with all fields; error types: `"timeout"`, `"api_error"`, `"validation_error"`

**MWMID-05** — Skill health alerts: `post/02-skill-health.py` tracks consecutive failures per skill; after 3 consecutive failures, writes alert to `~/.hermes/logs/skill-health-alerts.jsonl` (deduplicated per skill).
- Acceptance: 3 consecutive failures for skill `browser-use` → 1 alert; 4th failure → no second alert; success resets counter

**MWMID-06** — Configuration: `middleware:` section in `~/.hermes/config.yaml` with `enabled: true/false` and `skip: ["02-redact-secrets"]`.
- Acceptance: `enabled: false` skips all hooks; `skip` list excludes specified middleware

### Phase 51 — Agent Memory Client v2

**MEMCLI-01** — Vector-backed semantic search: `memory.search()` uses vector similarity (cosine distance) with relevance threshold (default 0.65); supports hybrid search (keyword + vector).
- Current: mem0 at `localhost:3201` provides basic keyword search
- Target: Semantic search returns relevant memories for queries with no exact keyword overlap
- Acceptance: Searching "how to configure models" returns config-related memories without exact keyword match; similarity scores in [0, 1]

**MEMCLI-02** — Relevance-scored context injection: At session start, top-N memories by relevance to session topic are injected instead of full `MEMORY.md`; capped at 1,500 characters.
- Acceptance: Session about "A/B tests" injects only A/B testing memories; total injected ≤ 1,500 chars; all injected memories have relevance ≥ 0.50

**MEMCLI-03** — Memory compression: New memory compared against existing; if cosine similarity > 0.90, merged into existing (union of content); `merge_count` tracks merges.
- Acceptance: Saving identical fact twice → 1 memory with `merge_count: 2`; saving similar but not identical facts → 1 merged memory

**MEMCLI-04** — Memory expiration: Each memory has `ttl_days` field (default: 90); expired memories purged by daily cron; moved to `~/.hermes/memory/archive/` (30-day grace).
- Acceptance: Memory with `ttl_days: 1` purged within 24h by cron; `ttl_days: 0` never expires; purged memories archived before deletion

**MEMCLI-05** — Backward compatibility: Existing `memory` tool API (`add`, `replace`, `remove`) works unchanged; missing fields get defaults on read.
- Acceptance: `memory(action='add', target='memory', content='test fact')` works as before; existing `MEMORY.md` imported into vector store on first run

### Phase 52 — Agent Observability Dashboard

**OBSUI-01** — Session timeline visualization: HTML dashboard showing sessions as timelines with tool call events, decisions, outputs.
- Current: No timeline — only Paperclip task status and hive posts
- Target: Static HTML page served locally; chronological view of last 20 sessions
- Acceptance: Open dashboard, see last 20 sessions, click into any session to see full timeline

**OBSUI-02** — Token usage tracking: Per-tool-call and per-session token counts from API response headers.
- Acceptance: Dashboard shows token totals per session and per tool type (web_search, browser, delegate_task, etc.)

**OBSUI-03** — Decision point logging: Agent logs "decision points" (tool choice, skill selection, clarification triggers) to structured JSON.
- Acceptance: Dashboard shows decision points on timeline with agent's stated reasoning

**OBSUI-04** — Error rate dashboard: Aggregate error rates by tool type over 1h/24h/7d windows.
- Acceptance: Dashboard shows error rate chart; clicking a spike reveals failing sessions

**OBSUI-05** — Session health score: Computed score (0-100) based on completion rate, error rate, token efficiency.
- Acceptance: Sessions with health < 50 flagged with reason (e.g., "3 tool failures in last hour")

## Explicit Non-Goals (v2.3)

- Cross-agent memory sharing — each agent has its own memory space
- Real-time streaming dashboard — post-session only for Phase 52
- OpenTelemetry integration at agent layer — emit to file; Memroos-side collectors consume later
- Modifying `lib/skill-budget.ts` — confirmed prompt-metadata tracker, not spend
- Multi-tenant RBAC — deferred to v2.4+ (aligns with v2.1 non-goals)
- Touching `lib/skill-budget.ts` for cost enforcement — category error
- Distributed tracing across agents — single-agent only for Phase 48
- Custom embedding models — use configured provider
- Skill marketplace / A/B testing — deferred further

## Out of Scope

| Item | Reason | Deferred To |
|------|--------|-------------|
| UI for middleware management | Read-only dashboard for Phase 52; management deferred | v2.4+ |
| Distributed middleware across agents | Single-agent only | v2.4+ |
| Real-time alerting | Alerts written to files; cron jobs consume | v2.4+ |
| Modifying tool return values | Post-call hooks observe but don't mutate | v2.4+ |
| Cross-agent memory sync | Session startup is the sync point | v2.4+ |
| Memory versioning/history | Merging replaces content, no undo | v2.4+ |
| External memory sources | Only `memory` tool store | v2.4+ |
| Mobile observability UI | Static HTML, offline only | v2.4+ |
| OTel SDK at agent layer | Emit to file; Memroos-side collectors consume later | v2.4+ |
