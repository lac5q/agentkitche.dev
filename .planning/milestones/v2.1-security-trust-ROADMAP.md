## Phase 42: Agent Shield + Iris Pre-flight Foundation

**Goal:** Establish the security baseline for public agent orchestration and ship the first Iris pre-flight gate.

**Requirements:** SEC2-01, SEC2-02, SEC2-03, SEC2-04, IRIS-01, IRIS-04, IRIS-05

**Success Criteria:**
1. Agent Shield scan results are documented and true/false-positive categories are separated for Memroos and Hermes.
2. Gaps beyond `content-scanner.ts` are documented.
3. `iris-scanner.ts` wraps existing `scanContent` coverage and adds prompt-injection pre-flight rules.
4. Dispatch UI/API and A2A task ingress call Iris before task persistence or delegation.
5. Blocked Iris decisions preserve existing external error and audit semantics.
6. Targeted Iris, dispatch, and A2A tests pass, and full Memroos tests/build/lint pass.

**Shipped:** `4d03fae` — `feat(iris): add pre-flight prompt injection scanner for dispatch and A2A`

## Phase 43: Tool Permission Guard + Policy Enforcement

**Goal:** Prevent agents from using tools, delegation targets, memory tiers, or network destinations outside policy.

**Requirements:** POLICY-01, POLICY-02, POLICY-03, POLICY-04, POLICY-05

**Success Criteria:**
1. Tool calls and A2A delegation attempts check canonical registry capabilities before execution.
2. Operating profile network policy blocks disallowed remote targets.
3. Memory writes enforce allowed tier and agent identity.
4. Denied actions return structured redacted errors.
5. Policy decisions are audited and queryable.
6. Existing Dispatch and A2A safe-path tests continue to pass.

**Execution starting points:**
- `apps/memroos/src/lib/agent-registry.ts` — canonical agent capabilities and metadata
- `apps/memroos/src/app/api/dispatch/route.ts` — manual Dispatch API ingress after Iris pre-flight
- `apps/memroos/src/lib/a2a/task-service.ts` — A2A task ingress after Iris pre-flight
- `apps/memroos/src/lib/a2a/config.ts` — operating profile and private-network policy
- `apps/memroos/src/lib/memory/tiers.ts` and `apps/memroos/src/app/api/memory/add/route.ts` — memory tier routing/enforcement
- `apps/memroos/src/lib/audit.ts` — policy decision audit sink

**Before editing code:** run GitNexus impact analysis for any target symbol being modified, especially `sendA2aMessage`, `POST` in `/api/dispatch`, `registerAgent`, and `resolveMemoryTier`.

**Local implementation notes:**
- Added shared policy helper at `apps/memroos/src/lib/security-policy.ts`.
- Dispatch policy now denies targets that declare capabilities but lack dispatch-compatible capabilities.
- A2A send policy now denies callers that declare capabilities but lack A2A send-compatible capabilities.
- Memory policy now denies tier writes outside declared memory capabilities.
- Legacy agents with no declared capabilities are allowed for backward compatibility; strict/permissive modes are deferred to Phase 45.
- Policy denials write `policy_denied` audit rows and return/throw structured policy errors without exposing secrets.
- Verified with targeted policy tests, affected A2A/dispatch/memory suites, full Memroos tests, lint, and production build.

## Phase 44: Security Operations UI + Reports

**Goal:** Make security posture visible and reviewable in Memroos.

**Requirements:** SECUI-01, SECUI-02, SECUI-03, SECUI-04

**Success Criteria:**
1. Security surface shows recent Iris, policy, secret-scan, and Agent Shield events.
2. Operators can filter by severity, agent, route, task, and action.
3. Security health summarizes open critical/high findings and last scan timestamps.
4. Release report generation redacts sensitive payloads.

**Local implementation notes:**
- Added `/api/security/report` with audit-derived security summaries, blocked-attempt counts, severity counts, control status, redacted timeline entries, and top actors.
- Added `SecurityOperationsPanel` to Library governance so operators can review security posture beside memory governance.
- Redaction covers bearer tokens, `sk-*` keys, `ak_*` keys, and common token/secret fields before dashboard display.
- Verified with route tests and browser checks on `/library`.

## Phase 45: Progressive Security Capability Exposure

**Goal:** Expose security capabilities through Memroos's progressive discovery model so agents and operators can see whether Iris/Agent Shield protections are available and which policy mode is active.

**Requirements:** SECCAP-01, SECCAP-02, SECCAP-03, SECCAP-04

**Success Criteria:**
1. Progressive tool/capability discovery exposes `capability:agent-shield` or equivalent Iris security capability.
2. Each registered agent can declare or inherit security mode: strict, standard, or permissive.
3. Security mode is visible in agent registry/detail surfaces without exposing secrets.
4. Capability loading and status checks degrade gracefully when Agent Shield is unavailable.

**Local implementation notes:**
- Added `/api/security/capabilities` with inherited default mode, per-agent strict/standard/permissive mode, readiness score, security capabilities, and enforced policy status.
- Added `AgentSecurityModesPanel` to the Agent Registry page.
- Agents can declare `metadata.securityMode`; otherwise they inherit `MEMROOS_SECURITY_MODE` with `standard` default.
- Verified with route tests and browser checks on `/agents`.
