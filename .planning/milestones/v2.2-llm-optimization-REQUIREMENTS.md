# Requirements: Agent Kitchen v2.2 — LLM Optimization + Evaluation

*Created: 2026-05-11*

## Goal

Make model choice data-driven across Agent Kitchen and connected agents. Agents should be able to ask Kitchen which model to use for a task, then report outcome data so recommendations improve over time.

## Requirements

### Phase 46 — Model Routing Telemetry Substrate

**LLMOPT-01** — Model outcome records: Store task type, agent, model, provider, strategy, cost estimate, latency, quality score, success/failure, error type, and context tags.

**LLMOPT-02** — Privacy-preserving defaults: Do not store raw prompt text by default; store hashes, redacted summaries, or explicit opt-in evidence only.

**LLMOPT-03** — Query API: Support model outcome queries by agent, task type, model, provider, outcome, and time window.

**LLMOPT-04** — Append API: Agents and services can append outcome records without changing A2A or REST task semantics.

### Phase 47 — Model Recommendation API + Knowledge Workspace

**LLMREC-01** — Recommendation endpoint: Return ranked model choices for a task type and preference strategy.

**LLMREC-02** — Strategy support: Support cost-first, quality-first, latency-first, and balanced recommendation modes.

**LLMREC-03** — Knowledge workspace: Maintain a synthesized model-selection guide that agents and humans can browse.

**LLMREC-04** — Cold-start fallback: Return sensible defaults and explain low confidence when historical data is sparse.

### Phase 48 — Evaluation Rigs + Quality Scoring

**EVAL-01** — Task-class eval sets: Define representative examples for core Kitchen/agent work types.

**EVAL-02** — Rubric scoring: Score outputs for correctness, completeness, safety, and usefulness.

**EVAL-03** — Regression checks: Detect when a routing change degrades quality or materially raises cost/latency.

**EVAL-04** — Recommendation feedback: Feed eval results into recommendation rankings.

### Phase 49 — Optimization Dashboard + Reports

**LLMUI-01** — Dashboard: Show model usage, cost, latency, quality, success rate, and drift by task class.

**LLMUI-02** — Explainability: Show why Kitchen recommends a model for a task.

**LLMUI-03** — Reports: Generate summaries of cheaper/faster substitutions and quality risks.

**LLMUI-04** — Privacy: Do not display raw private prompts by default.

## Non-Goals

- Replacing provider SDKs or creating a universal LLM proxy in v2.2.
- Fine-tuning models.
- Optimizing prompt storage by keeping raw private prompts.
- Caching and cold-start performance work; that is v2.4.
