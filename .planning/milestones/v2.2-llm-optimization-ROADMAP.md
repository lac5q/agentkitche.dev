## Phase 46: Model Routing Telemetry Substrate

**Goal:** Persist enough structured data to learn which models work best for each task type while avoiding raw prompt storage by default.

**Requirements:** LLMOPT-01, LLMOPT-02, LLMOPT-03, LLMOPT-04

**Success Criteria:**
1. Each record captures task type, agent, model, provider, cost estimate, latency, success/failure, quality score, and context tags.
2. Raw prompt text is omitted by default; stored evidence is redacted or hashed.
3. Records are queryable by agent, task type, provider/model, outcome, and time window.
4. Existing agents can append records through a simple API.

**Local implementation notes:**
- Added `model_routing_events` schema behind `ensureModelRoutingSchema()` to avoid changing global startup schema.
- Added `/api/model-routing/telemetry` for GET/POST telemetry.
- Prompt text is never returned or stored; optional prompt input is hashed with SHA-256.

## Phase 47: Model Recommendation API + Knowledge Workspace

**Goal:** Provide a recommendation path agents can query before choosing a model, with outcome append after completion.

**Requirements:** LLMREC-01, LLMREC-02, LLMREC-03, LLMREC-04

**Success Criteria:**
1. Recommendation endpoint returns ranked model choices with rationale.
2. Callers can choose cost-first, quality-first, latency-first, or balanced strategy.
3. Knowledge workspace synthesizes a browsable model-selection guide.
4. Cold-start recommendations degrade gracefully when history is sparse.

**Local implementation notes:**
- Added `/api/model-routing/recommendations` with balanced, cost, quality, and latency strategies.
- Recommendations combine a cold-start model catalog with observed telemetry outcomes.
- Added client hooks so agents and UI can consume the recommendation loop.

## Phase 48: Evaluation Rigs + Quality Scoring

**Goal:** Add repeatable quality measurement so recommendations do not optimize for cheap failures.

**Requirements:** EVAL-01, EVAL-02, EVAL-03, EVAL-04

**Success Criteria:**
1. Core task classes have reference examples and scoring rubrics.
2. Candidate models can be compared on quality, cost, latency, and failure mode.
3. Regression checks catch degraded model choices.
4. Eval scores feed recommendation rankings.

**Local implementation notes:**
- Added `/api/model-routing/evals` with task-fit, knowledge-use, cost-latency, and outcome-quality rubrics.
- Added reference tasks for product, sales, and engineering task classes.
- Eval summaries draw from telemetry so recommendation quality can be monitored over time.

## Phase 49: Optimization Dashboard + Reports

**Goal:** Make model performance and routing choices visible and actionable in Kitchen.

**Requirements:** LLMUI-01, LLMUI-02, LLMUI-03, LLMUI-04

**Success Criteria:**
1. Dashboard shows usage, cost, latency, quality, success rate, and drift by task class.
2. Operators can inspect why a model is recommended.
3. Reports identify cheaper/faster substitutions that preserve quality.
4. UI avoids raw private prompt display by default.

**Local implementation notes:**
- Added `ModelRoutingPanel` to Usage with telemetry KPIs, strategy controls, ranked model recommendations, and eval dimensions.
- Dashboard shows model score, observation count, quality, success, and latency without exposing raw prompts.
- Verified with route tests and browser checks on `/ledger`.
