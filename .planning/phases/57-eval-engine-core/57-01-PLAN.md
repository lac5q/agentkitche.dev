# Phase 57 Plan: Eval Engine Core

## Product Goal

Give MemroOS one auditable eval signal, `W`, that can score any registered agent trace before later self-improvement loops decide whether to keep or discard a change.

## Requirements Covered

- EVAL-01: Scorer registry
- EVAL-02: 3-layer composite scalar `W`
- EVAL-03: pinned cross-family judge metadata
- EVAL-04: golden-set framework
- EVAL-05: drift guard
- EVAL-06: position-bias swap and self-preference mitigation
- EVAL-07: `memroos.eval.yaml`
- EVAL-08: Kitchen UI eval panel
- EVAL-09: memory recall as registered scorer
- EVAL-10: eval-run history persistence

## Tasks

1. Add RED tests for config loading, scorer registration, composite `W`, drift guard, memory scorer adapter, persistence, and API routes.
2. Implement eval engine library modules under `apps/kitchen/src/lib/evals/`.
3. Add additive SQLite schema for `eval_runs` and `eval_run_examples`.
4. Add root `memroos.eval.yaml` and seed golden sets.
5. Add `/api/evals/config`, `/api/evals/run`, and `/api/evals/history`.
6. Add Kitchen UI panel and navigation entry.
7. Update planning docs and requirements checkboxes for Phase 57 only.
8. Run targeted tests, lint/build as budget allows, GitNexus detect changes, then commit.

## Risk Notes

- `initSchema` impact is CRITICAL because every DB-backed route initializes through it. Mitigation: additive-only DDL and tests.
- Judge calls must not leak prompt text to external providers in Phase 57. Mitigation: deterministic local judge adapter with prompt hash and pinned metadata only.
- Same-family agent/judge pairing must block trusted `W`. Mitigation: explicit family comparison in engine tests.
