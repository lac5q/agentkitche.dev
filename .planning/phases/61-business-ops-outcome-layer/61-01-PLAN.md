# Phase 61 Plan: Business-Ops Outcome Layer (L3)

## Product Goal

Close the L3 layer of composite `W` with real business signal: post-hoc outcome events pulled from CRM, helpdesk, and finance systems flow into a `business_outcome_events` store, five L3 scorers read from that store synchronously, per-company KPI weighting is config-driven, and a `/business-ops` dashboard shows operators how each agent is performing against business outcomes over time.

## Requirements Covered

- L3-01: Trace post-hoc metric extraction — completion rate, escalation rate, TTR p50, operator approval rate, cost-per-task
- L3-02: CRM adapter (HubSpot live + Salesforce fixture)
- L3-03: Helpdesk adapter (Intercom live + Zendesk fixture)
- L3-04: Finance-system adapter (QuickBooks live + NetSuite fixture)
- L3-05: Per-company KPI weighting in `memroos.eval.yaml` — no code change required to tune
- L3-06: Business-ops dashboard — per-agent W timeline with L1/L2/L3 breakdown and click-through to source trace and business-system record

3. Add `business_outcome_events` table (additive only) to `initSchema` in `apps/kitchen/src/lib/db-schema.ts`:
   ```sql
   CREATE TABLE IF NOT EXISTS business_outcome_events (
     id              INTEGER PRIMARY KEY,
     correlation_id  TEXT    NOT NULL,
     agent_id        TEXT,
     company_id      TEXT,
     tenant_id       TEXT    NOT NULL DEFAULT 'default-tenant',
     source          TEXT    NOT NULL CHECK(source IN ('crm','helpdesk','finance')),
     adapter         TEXT    NOT NULL,
     event_type      TEXT    NOT NULL,
     payload_json    TEXT    NOT NULL,
     source_url      TEXT,
     observed_at     TEXT    NOT NULL,
     created_at      TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now'))
   );
   CREATE INDEX IF NOT EXISTS boe_correlation ON business_outcome_events(correlation_id);
   CREATE INDEX IF NOT EXISTS boe_agent       ON business_outcome_events(agent_id, observed_at DESC);
   CREATE INDEX IF NOT EXISTS boe_tenant      ON business_outcome_events(tenant_id, observed_at DESC);
   ```

4. Implement business-ops library modules under `apps/kitchen/src/lib/business-ops/`:
   - `adapter-interface.ts` — `BusinessSystemAdapter` interface (`name: string`, `category: BusinessOutcomeCategory`, `pull(correlationIds: string[]): Promise<BusinessOutcomeEvent[]>`), `BusinessOutcomeEvent` type (`id`, `correlationId`, `agentId`, `companyId`, `source`, `adapter`, `eventType`, `payloadJson`, `sourceUrl?`, `observedAt`), `BusinessOutcomeCategory` enum
   - `event-store.ts` — `writeOutcomeEvents(db, events: BusinessOutcomeEvent[]): void` (upsert by `(correlation_id, adapter, event_type, observed_at)` to avoid duplicates on retry), `getEventsForCorrelationId(db, correlationId): BusinessOutcomeEvent[]`, `getEventsForAgent(db, agentId, since?): BusinessOutcomeEvent[]`
   - `extractor.ts` — `extractKpis(trace: AgentEvalTrace, events: BusinessOutcomeEvent[]): CanonicalKpiSet` where `CanonicalKpiSet = { completionRate: number | null, escalationRate: number | null, ttrP50Ms: number | null, operatorApprovalRate: number | null, costUsd: number | null }`
   - `l3-scorers.ts` — five `EvalScorer` implementations: `completion_rate_scorer`, `escalation_rate_scorer`, `ttr_p50_scorer`, `operator_approval_scorer`, `cost_per_task_scorer`. Each calls `getEventsForCorrelationId()` via the injected db. Returns `EvalScorerResult` with `score: 0` and `metadata.unavailable: true` when no events exist, so the engine can detect null-L3.
   - `adapter-runner.ts` — `runAdapterPoll(db, correlationIds: string[], adapters: BusinessSystemAdapter[]): Promise<{eventsWritten: number, errors: string[]}>` — iterates adapters, calls `pull()`, calls `writeOutcomeEvents()`

## Risk Notes

- **`initSchema` impact is CRITICAL** — every DB-backed route initializes through it. Mitigation: run `gitnexus_impact` first (task 1), use additive-only DDL (`CREATE TABLE IF NOT EXISTS`), no `ALTER` or `DROP` on existing tables, new table added at end of function.
- **Null-L3 in composite W** — if the engine does not handle null L3, all in-flight tasks that have not yet received business-system events will score incorrectly low. Mitigation: implement and test the null-L3 renormalization path before wiring L3 scorers into production (task 7 precedes task 8).
- **Fixture adapter leakage into production** — mock fixture adapters must not run in production. Mitigation: runtime guard `if (process.env.NODE_ENV === 'production') return []` + explicit `@fixture` JSDoc tag + excluded from adapter discovery in `adapter-runner.ts` (only non-fixture adapters are registered by default).
- **Salesforce/Zendesk/NetSuite deferred scope** — six production adapters is over-scoped for one phase. Only HubSpot, Intercom, QuickBooks are live v1; the remaining three are mock fixtures. This is a deliberate scope trim; the Roadmap success criteria for L3-02/03/04 are partially satisfied (signals flow end-to-end via the live adapters; the fixture adapters prove the interface contract). Note in the commit message.
- **External API credentials in CI** — live adapter tests against real APIs must not run in CI without credentials. Mitigation: unit-test each adapter against canned HTTP fixture responses (using `msw` or equivalent); integration tests that call real APIs are gated by a `BUSINESS_OPS_INTEGRATION_TESTS=true` env flag.
- **Sub-weight validation** — `companies:` config block sub-weights must sum to 1.0. Mitigation: config reader throws with a clear actionable error at startup if invalid; covered by test in task 2.
