# Phase 62 Plan: Public Eval API + SDK

## Product Goal

Open MemroOS's eval engine to the world: a versioned, tenant-isolated HTTP surface accepts any agent trace and returns `W` + layer breakdown + proposal queue; TypeScript and Python SDKs wrap the surface with idiomatic ergonomics; Phase 59 and Phase 60's autogen loops are refactored to route through the public API (proving the framework-agnostic contract on real internal traffic); and a customer quickstart gets a new user to their first scored trace in five minutes.

## Requirements Covered

- API-01: Framework-agnostic HTTP eval surface — customer's agent traces in, W + layer breakdown + proposal queue out
- API-02: OpenInference trace format support (pinned to openinference-semantic-conventions v0.1.x) + MemroOS JSON format (`AgentEvalTrace`)
- API-03: TypeScript (`@memroos/sdk`) and Python (`memroos`) SDKs with idiomatic ergonomics and smoke-tests
- API-04: Per-customer API keys enforce tenant isolation; no cross-tenant trace, golden set, or proposal visibility; verified by test
- API-05: Customer-facing quickstart — "first eval in 5 minutes" from `pip install memroos` / `npm i @memroos/sdk` through first scored trace
- API-06: Phase 59 memory autogen loop + Phase 60 agent autogen loop use the public API surface end-to-end; not an internal-only path

7. Add public API routes under `apps/kitchen/src/app/api/public/v1/`:

   `eval/route.ts` — POST:
   - Extract `Authorization: Bearer <key>` header; call `resolveTenant()`; return 401 if null
   - Call `checkRateLimit()`; return 429 with `X-RateLimit-*` headers if not allowed
   - Detect format: call `isOpenInferenceTrace(body)`; map with `mapOpenInferenceToAgentEvalTrace()` if true; otherwise parse as `AgentEvalTrace`
   - Inject `tenantId` into trace metadata and pass to `scoreAndMaybePersistEvalTrace()`
   - Return `{runId, w: result.compositeW, layers: result.layers, proposalIds: []}` (proposal IDs populated in future; empty array at v1 until SEAL integration is hooked)
   - Add `X-RateLimit-*` headers to success response

   `eval/[runId]/route.ts` — GET:
   - Resolve tenant; fetch run by ID; verify `run.tenantId === resolvedTenantId`; return 403 on mismatch; return `EvalRunResult` on match

   `proposals/route.ts` — GET:
   - Resolve tenant; query `seal_proposals WHERE tenant_id = ?`; optional `?traceId=` filter; return array of `{id, proposalType, status, forecastWDelta, createdAt}`

   `proposals/[proposalId]/route.ts` — GET:
   - Resolve tenant; fetch proposal; verify `proposal.tenantId === resolvedTenantId`; return 403 on mismatch; return full proposal including `diffJson` and `baselineLayerJson`

   All routes: `export const dynamic = "force-dynamic"` per Kitchen convention.

## Risk Notes

- **`initSchema` impact is CRITICAL** — `ALTER TABLE ADD COLUMN` on eight tables runs at every cold start. Mitigation: wrap each `ALTER TABLE` in a try/catch that silently ignores `"duplicate column name"` errors (SQLite does not support `ADD COLUMN IF NOT EXISTS`). This is the standard pattern for additive migrations in this codebase — apply the same guard.
- **Dogfood refactor breaks SEAL apply loop if SDK is unavailable** — if the Kitchen server is not running when `apply.ts` tries the localhost SDK call, the SEAL loop fails. Mitigation: wrap the SDK call in a try/catch; on connection error, log a warning and fall back to a direct `EvalService.runForTrace()` call in development only (`NODE_ENV !== 'production'`). Document this fallback as a development convenience, not production behavior.
- **Cross-tenant leakage in SQL queries** — every query against v2.5 tables that previously omitted `WHERE tenant_id = ?` must now include it. Mitigation: grep for all usages of `eval_runs`, `seal_proposals`, etc. in query strings after the migration, and verify each includes the tenant scope. Cover each public API route with a cross-tenant isolation test (task 2).
- **SQLite `ALTER TABLE` idempotency** — re-running `initSchema` on an existing DB would fail with "duplicate column name". Mitigation: wrap each `ALTER TABLE` in try/catch ignoring that specific error. Test by running `initSchema` twice against the same in-memory DB in a test.
- **SDK published types diverge from internal types** — `AgentEvalTrace` is copied (not workspace-imported) into `packages/sdk-ts/src/types.ts` to make the SDK standalone-publishable. If the internal type changes, the SDK copy must be updated manually. Mitigation: add a CI step that compares the SDK `AgentEvalTrace` fields against the internal type and fails if they diverge (a simple JSON schema comparison or a TypeScript `Satisfies` check in a test file).
- **Phase 59/60 not yet shipped when Phase 62 is authored** — the dogfood refactor in task 9 targets modules from Phases 58/59/60. If those phases have not landed by the time Phase 62 executes, the refactor targets `apps/kitchen/src/lib/seal/apply.ts` (Phase 58, confirmed shipped) and stubs the Phase 59/60 callers with a TODO pointing to this plan. The integration test (task 2) gates on both phases being present.
- **Rate limiter in-memory state across Next.js restarts** — in-memory LRU resets on every restart; rate limit windows do not persist. This is acceptable for v1 single-instance; document in `rate-limiter.ts` with a `// v2: replace with Redis-backed store for multi-instance` comment.

## Amendment 2026-05-16 — As-built routes ratified

The original plan specified `/v1/eval`-style routes; the as-built public API
uses `/api/public/v1/{traces,runs,proposals}`. **Decision: ratify the as-built
routes.** Rationale: code is green and tested, the SDKs target the as-built
paths, and external product packaging (the point at which a frozen public
contract matters) is explicitly deferred per
`.planning/seeds/eval-engine-as-product.md`. The public route vocabulary will be
finalized as part of that external-packaging effort, not retrofitted now.
