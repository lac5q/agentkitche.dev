# Phase 60 Plan: Agent Autogen Learnings

## Product Goal

Extend the Phase 58 SEAL substrate with three agent-level proposal types so MemroOS can self-improve agent behavior — not just memory policy. Four role-specific golden sets (sales, support, finance, ops) ship fully populated and human-validated, trajectory evals score multi-step agent runs against the L2 layer, and three named weight preset profiles let operators adopt the autogen loop without manual weight tuning. The loop is demonstrated end-to-end on dogfood data, producing at least one approved proposal that lifts composite W.

## Requirements Covered

- AGENTGEN-01: `agent_instruction_patch` — propose edits to an agent's system prompt / operating instructions
- AGENTGEN-02: `skill_addition` — propose new skill registration when observed gaps repeat
- AGENTGEN-03: `tool_routing_update` — propose changes to which tool an agent reaches for in a given context
- AGENTGEN-04: Per-role golden sets — sales / support / finance / ops, ~50 examples each, in `golden-sets/`
- AGENTGEN-05: Trajectory evals (Phoenix-style) — score full multi-step agent runs, not single-turn outputs
- AGENTGEN-06: Named preset weight profiles — `outcome-weighted`, `quality-weighted`, `compliance-weighted`

### Task 2 — Schema additions (additive only)

Add three tables to `initSchema` in `apps/kitchen/src/lib/db-schema.ts`. All are `CREATE TABLE IF NOT EXISTS` — no ALTER or DROP on existing tables.

```sql
-- Mutation target for agent_instruction_patch
CREATE TABLE IF NOT EXISTS agent_instructions (
  id           INTEGER PRIMARY KEY,
  agent_id     TEXT    NOT NULL REFERENCES registered_agents(id) ON DELETE CASCADE,
  instructions_text TEXT NOT NULL,
  version      INTEGER NOT NULL DEFAULT 1,
  proposal_id  TEXT    REFERENCES seal_proposals(id),
  is_active    INTEGER NOT NULL DEFAULT 1 CHECK(is_active IN (0,1)),
  created_at   TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now'))
);
CREATE INDEX IF NOT EXISTS agent_instructions_active
  ON agent_instructions(agent_id, is_active, version DESC);

-- Mutation staging/promotion target for skill_addition
CREATE TABLE IF NOT EXISTS proposed_skills (
  id           INTEGER PRIMARY KEY,
  agent_id     TEXT    NOT NULL REFERENCES registered_agents(id) ON DELETE CASCADE,
  skill_id     TEXT    NOT NULL,
  action       TEXT    NOT NULL,
  metadata     TEXT    NOT NULL DEFAULT '{}',
  proposal_id  TEXT    NOT NULL REFERENCES seal_proposals(id),
  status       TEXT    NOT NULL DEFAULT 'proposed'
               CHECK(status IN ('proposed','promoted','rolled_back')),
  created_at   TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now'))
);
CREATE INDEX IF NOT EXISTS proposed_skills_proposal
  ON proposed_skills(proposal_id, status);

-- Mutation target for tool_routing_update
CREATE TABLE IF NOT EXISTS agent_tool_routing_policies (
  id               INTEGER PRIMARY KEY,
  agent_id         TEXT    NOT NULL REFERENCES registered_agents(id) ON DELETE CASCADE,
  tool_name        TEXT    NOT NULL,
  context_pattern  TEXT    NOT NULL DEFAULT '*',
  preference_weight REAL   NOT NULL DEFAULT 1.0,
  proposal_id      TEXT    REFERENCES seal_proposals(id),
  is_active        INTEGER NOT NULL DEFAULT 1 CHECK(is_active IN (0,1)),
  created_at       TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now'))
);
CREATE INDEX IF NOT EXISTS agent_tool_routing_active
  ON agent_tool_routing_policies(agent_id, tool_name, is_active);
```

### Task 3 — Extend proposal registry

Edit `apps/kitchen/src/lib/seal/proposal-registry.ts`. Add three new entries to the `PROPOSAL_TYPES` const — do not create a new registry file. Each entry implements `ProposalRegistryEntry`: `type`, `label`, `description`, `buildDraft()`, `applyShadow()`.

```
agent_instruction_patch:
  label: "Agent instruction patch"
  description: "Propose an edit to the agent's system prompt or operating instructions."
  buildDraft: reads current active agent_instructions row for the agent; produces a text diff in diff_json
  applyShadow: writes a new agent_instructions row (is_active=1) and flips previous row (is_active=0) inside a transaction

skill_addition:
  label: "Skill addition"
  description: "Propose registering a new skill for the agent when observed gaps repeat."
  buildDraft: produces a structured proposed skill record in diff_json
  applyShadow: inserts into proposed_skills; on keep, copies to agent_skill_reports

tool_routing_update:
  label: "Tool routing update"
  description: "Propose a change to the agent's tool preference weight for a given context pattern."
  buildDraft: produces {tool_name, context_pattern, old_weight, new_weight} in diff_json
  applyShadow: writes a new agent_tool_routing_policies row, flips old row is_active=0
```

The `// v2+ extension path` comment block is preserved unchanged.

After adding the three entries, the TypeScript `ProposalType` union automatically expands to include all three new keys (it is derived from `keyof typeof PROPOSAL_TYPES`). No manual union editing required.

## Risk Notes

- **`initSchema` is CRITICAL blast radius.** All DB-backed routes initialize through it. The three new tables are additive-only (`CREATE TABLE IF NOT EXISTS`, no ALTER/DROP). Mitigation: add at end of function, wrap in transactions, run full test suite post-schema-add.

- **`agent_instructions` table pre-condition.** If any existing agent reads instructions from `registered_agents.metadata`, the `agent_instruction_patch` apply step will not affect that agent. Mitigation: grep `metadata` usage for `system_prompt` or `instructions` keys before marking AGENTGEN-01 complete.

- **Golden set expansion quality.** 50-example golden sets that are too narrow (all single-turn, all "completed: true") will fail to stress the trajectory scorer or the autogen loop meaningfully. Mitigation: ensure at least 10 examples per role have `steps` arrays (multi-step), at least 5 have `escalated: true`, and at least 5 have failing tool calls — to exercise the drift guard and trajectory penalty logic.

- **Trajectory scorer L2 dilution.** Adding `trajectory_multi_step` to L2 changes W values for all existing traces. Existing golden-set human scores were calibrated against the old scorer list. Mitigation: after adding the scorer, rerun the drift guard on `business-ops-50.jsonl` and confirm agreement ≥ 0.85 with the new scorer list before using W in the autogen loop.

- **Phase 57 `config.ts` write path.** Phase 57 may not have implemented a disk-write path for `memroos.eval.yaml` (only a read path). `setActivePreset()` requires writing YAML. Mitigation: if `config.ts` has no write path, implement a minimal YAML patch writer in `presets.ts` that reads the file, modifies the `active_preset` key, and writes back without reformatting other keys.
