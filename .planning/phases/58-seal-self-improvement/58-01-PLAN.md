# Phase 58 Plan: SEAL Self-Improvement Substrate

## Product Goal

Give MemroOS a generic, auditable self-improvement loop: reflection on any low-W agent trace produces typed proposals; operators approve or reject from a Kitchen UI queue; approved proposals are applied in an isolated context, evals rerun, and the mutation kept only if composite W improves over the persisted baseline; regressed mutations are rolled back automatically with every decision written to an immutable audit log.

## Requirements Covered

- SEAL-01: Reflection step — given trace + W breakdown, generate one or more typed proposals (registered proposal types only)
- SEAL-02: Polymorphic proposal-type registry — pluggable mutation surfaces register against one interface (closed at v1)
- SEAL-03: Operator approval queue — Kitchen UI surfaces pending proposals with W-delta forecast, diff, and approve/reject controls
- SEAL-04: Apply → rerun-evals → keep-if-W-improved cycle — proposal applied in isolated context, evals rerun against same golden set, kept only if composite W ≥ pre-mutation baseline
- SEAL-05: Rollback with full audit trail — failed proposals reverted, every decision persisted with reasoning
- SEAL-06: Proposal-type registry is a closed enum at v1; extension path documented

## Risk Notes

- `initSchema` impact is CRITICAL — every DB-backed route initializes through it. Mitigation: additive-only DDL (no ALTER/DROP on existing tables), Phase 58 tables added at end of function, all three tables wrapped in `CREATE TABLE IF NOT EXISTS`.
- Isolated-apply leakage: the shadow-copy strategy must never commit partial state to the main resource if the eval rerun fails or returns W_post < W_baseline. Mitigation: apply.ts wraps all mutation writes in a SQLite transaction that is rolled back on failure; for text-payload mutations (v1 only) no file-system state is touched until after the eval rerun passes.
- Audit log mutability: `seal_audit_log` must have no UPDATE or DELETE routes. Mitigation: audit.ts exports no update/delete functions; API routes expose GET only for `/api/seal/audit`; enforce in review checklist.
- Closed registry vs plugin temptation: downstream phases (59, 60) populate the registry via registry-update commits, not runtime registration. Mitigation: the `ProposalType` union and exhaustiveness check make it a TypeScript compile error to reference an unregistered type; document this explicitly in `proposal-registry.ts`.
- Phase 57 dependency: `apply.ts` calls `EvalService.runForTrace()` to rerun evals post-apply. If Phase 57 does not expose this interface, Phase 58 must stub it. The plan assumes Phase 57 ships `EvalService` with at least `getRunById()` and `runForTrace(traceId, agentId, goldenSetPath)`.
