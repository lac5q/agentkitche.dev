```typescript
//   dev tool i:   { x: 20 + i * DEV_TOOL_SPACING,                   y: DEV_TOOL_Y }    = { x: 20  + i*140, y: 740 }

// Parent-relative offsets after migration (child.pos = child_absolute - parent_absolute):
//   agent-{id}:   parentId:"group-agents",   position: { x: 15 + i * AGENT_SPACING,    y: 32 }
//   local-agents: parentId:"group-agents",   position: { x: 15 + keyRemote.length * AGENT_SPACING, y: 32 }
//   dev tool i:   parentId:"group-devtools", position: { x: 15 + i * DEV_TOOL_SPACING, y: 32 }
// Every child also gets: extent: "parent"
```

React Flow v12 parentId mechanics (from @xyflow/react):
- `parentId: string` — declares the parent group node's id; child position becomes relative to parent
- `extent: "parent"` — keeps child within parent bounds during drag
- CRITICAL: parent MUST appear before children in the nodes array or React Flow logs a warning and positions are wrong on first render
- Group box nodes KEEP absolute position; they are roots (no parentId of their own)
- `zIndex: -1` on group boxes MUST stay (they render behind children)

Static nodes that are NOT children of any group (stay absolute, do NOT get parentId):
  request, gateways, manager, output, tunnels, taskboard, notebooks, librarian,
  qdrant, cookbooks, apo, gitnexus, llmwiki, knowledge-curator, obsidian
