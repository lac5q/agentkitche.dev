<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| filesystem → /api/skills route | `failures.log` content is machine-generated; route must not crash or expose raw content in response. |
| /api/skills → browser | Response is public to anyone on the LAN/tunnel — must not echo traceback text or sensitive paths. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-05 | Information Disclosure | /api/skills response | mitigate | Response includes ONLY aggregate counts keyed by `agent_id` and `error_type` — NOT raw entries, NOT tracebacks, NOT filesystem paths. Aggregator strips everything else. |
| T-14-06 | Denial of Service | Route reading large failures.log | mitigate | Parser uses single `readFile` — bounded by OS-level log rotation. If file grows unbounded, route response time degrades but does not crash. Accept risk for now; document in SUMMARY that rotation is assumed. |
| T-14-07 | Tampering | Third-party writes to failures.log | mitigate | Parser skips malformed entries instead of failing entire aggregation — a poisoned entry cannot deny aggregate data to the dashboard. |
| T-14-08 | Elevation of Privilege | execSync/exec in new code | mitigate | Zero shell execution — only `fs/promises.readFile`. Pre-commit hook enforces this; code review confirms. |
| T-14-09 | Information Disclosure | UI rendering traceback | accept | UI only renders counts and top error_type enum label — never raw entries. Confirmed by Task 3's optional-chaining rules. |
</threat_model>
