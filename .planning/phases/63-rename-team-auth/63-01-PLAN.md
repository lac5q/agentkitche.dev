# Phase 63 Plan 01: Rename + Team Auth

## Product Goal

Memroos is renamed to Memroos everywhere a user or contributor can see it. The platform supports multiple authenticated human users with role-based access (admin / operator / reviewer), JWT-based login with httpOnly refresh cookies, per-user API keys, and an invite-link flow for adding team members. Phase 64 (Immutable Audit) depends on the `user_id` actor concept this phase introduces.

## Requirements Covered

- RENAME-01: Product rename Memroos → Memroos across codebase, docs, config, UI, and package names
- TEAM-01: Role-based access control — admin, operator, reviewer roles enforced via middleware
- TEAM-02: Multi-user session auth — JWT access token + refresh cookie, per-user API keys, team invitation
- TEAM-03: Operator approval queue wired to authenticated user identity

```sql
-- users: human team members
CREATE TABLE IF NOT EXISTS users (
  id            TEXT PRIMARY KEY,
  email         TEXT NOT NULL UNIQUE,
  display_name  TEXT NOT NULL,
  password_hash TEXT NOT NULL,
  tenant_id     TEXT NOT NULL DEFAULT 'default-tenant'
                REFERENCES tenants(id) ON DELETE CASCADE,
  created_at    TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now')),
  last_login_at TEXT
);
CREATE INDEX IF NOT EXISTS users_email ON users(email);
CREATE INDEX IF NOT EXISTS users_tenant ON users(tenant_id);

-- user_roles: role assignments (one user can have multiple roles)
CREATE TABLE IF NOT EXISTS user_roles (
  user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  role    TEXT NOT NULL CHECK(role IN ('admin','operator','reviewer')),
  PRIMARY KEY (user_id, role)
);

-- user_api_keys: per-user programmatic access keys
CREATE TABLE IF NOT EXISTS user_api_keys (
  id           TEXT PRIMARY KEY,
  user_id      TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  key_hash     TEXT NOT NULL UNIQUE,
  label        TEXT NOT NULL DEFAULT '',
  created_at   TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now')),
  last_used_at TEXT,
  revoked_at   TEXT
);
CREATE INDEX IF NOT EXISTS uak_user ON user_api_keys(user_id, revoked_at);
CREATE INDEX IF NOT EXISTS uak_hash ON user_api_keys(key_hash);

-- user_refresh_tokens: httpOnly refresh token store
CREATE TABLE IF NOT EXISTS user_refresh_tokens (
  id         TEXT PRIMARY KEY,
  user_id    TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  token_hash TEXT NOT NULL UNIQUE,
  expires_at TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now')),
  revoked_at TEXT
);
CREATE INDEX IF NOT EXISTS urt_user ON user_refresh_tokens(user_id, revoked_at);
CREATE INDEX IF NOT EXISTS urt_hash ON user_refresh_tokens(token_hash);

-- team_invitations: single-use invite link tokens
CREATE TABLE IF NOT EXISTS team_invitations (
  id         TEXT PRIMARY KEY,
  token_hash TEXT NOT NULL UNIQUE,
  role       TEXT NOT NULL CHECK(role IN ('admin','operator','reviewer')),
  invited_by TEXT NOT NULL REFERENCES users(id),
  email_hint TEXT,
  used_at    TEXT,
  expires_at TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ','now'))
);
CREATE INDEX IF NOT EXISTS inv_token ON team_invitations(token_hash, used_at);
```

### Part C — Auth Library (TEAM-02)

**C1. New file: `apps/memroos/src/lib/auth/jwt.ts`**

Exports:
- `signAccessToken(payload: { sub: string; roles: string[] }): Promise<string>` — signs HS256 JWT, 15-minute expiry, using `MEMROOS_JWT_SECRET` env var via `jose`
- `verifyAccessToken(token: string): Promise<{ sub: string; roles: string[] } | null>` — verifies signature and expiry; returns null on any error
- `MEMROOS_JWT_SECRET` read from `process.env.MEMROOS_JWT_SECRET` — throw at startup if absent

**C2. New file: `apps/memroos/src/lib/auth/user-auth.ts`**

Exports:
- `UserContext` interface: `{ userId: string; roles: string[]; tenantId: string }`
- `authenticateUser(req: Request, db?: Database): Promise<UserContext | null>`
  - First checks `Authorization: Bearer <token>` header
  - If token parses as a JWT (contains two `.` separators): verify with `verifyAccessToken`; look up user record for `sub`; return `UserContext`
  - If token does not look like a JWT: treat as user API key; hash with SHA-256; look up `user_api_keys`; check `revoked_at`; look up user + roles; return `UserContext`
  - Returns `null` on any failure
- `requireRole(ctx: UserContext | null, minRole: 'reviewer' | 'operator' | 'admin'): boolean` — returns true if context has at least the specified role (role hierarchy: admin > operator > reviewer)

**C3. New file: `apps/memroos/src/lib/auth/passwords.ts`**

Exports:
- `hashPassword(plain: string): Promise<string>` — `bcryptjs.hash(plain, 12)`
- `verifyPassword(plain: string, hash: string): Promise<boolean>` — `bcryptjs.compare(plain, hash)`

**C4. New file: `apps/memroos/src/lib/auth/invitations.ts`**

Exports:
- `createInvitation(db, { role, invitedBy, emailHint? }): { inviteUrl: string; tokenRaw: string }` — generates 32-byte random token, stores hash, returns raw token and full URL
- `consumeInvitation(db, tokenRaw: string): { role: string; emailHint?: string } | null` — validates token, checks expiry, checks not used; does NOT mark used (caller does that on successful registration)
- `markInvitationUsed(db, tokenRaw: string): void`

### Part D — Auth API Routes (TEAM-02)

All routes under `apps/memroos/src/app/api/auth/`.

**D1. `POST /api/auth/login` — `route.ts`**

Request body: `{ email: string; password: string }`
- Look up user by email
- Verify password with `verifyPassword`
- On success: `signAccessToken({ sub: userId, roles })`, generate refresh token (32-byte random), store hash in `user_refresh_tokens`, set `memroos_refresh` cookie (httpOnly, SameSite=Lax, Secure in prod)
- Response: `{ accessToken: string; user: { id, email, displayName, roles } }`
- On failure: 401, no detail about which field was wrong

**D2. `POST /api/auth/refresh` — `route.ts`**

- Read `memroos_refresh` cookie from request
- Hash and look up in `user_refresh_tokens`; check `revoked_at` and `expires_at`
- Revoke old token, insert new refresh token (rotation)
- Return new access token in response body; set new refresh cookie
- On failure: 401, clear cookie

**D3. `POST /api/auth/logout` — `route.ts`**

- Read `memroos_refresh` cookie; if present, revoke the token in DB
- Clear `memroos_refresh` cookie (set Max-Age=0)
- Return 200

**D4. `GET /api/auth/invite/[token]` — `route.ts`**

- `consumeInvitation(db, token)` (read-only check, does not mark used)
- If valid: `200 { role, emailHint }`
- If invalid/expired/used: `404 { error: 'invalid or expired invitation' }`

**D5. `POST /api/auth/invite/[token]` — `route.ts`**

Request body: `{ email: string; displayName: string; password: string }`
- `consumeInvitation` to validate token
- Check email not already registered
- `hashPassword`, insert into `users`, insert into `user_roles`
- `markInvitationUsed`
- Return `201 { userId }`
- On any error: appropriate 4xx

**D6. `POST /api/auth/invite` (admin only) — `route.ts`**

Request body: `{ role: string; emailHint?: string }`
- Require admin role (check JWT, return 403 if not admin)
- `createInvitation(db, { role, invitedBy: userId, emailHint })`
- Return `201 { inviteUrl }`

**D7. `GET /api/users` (admin only) — `route.ts`**

- Require admin role
- Return list of users with roles (no password hashes)

**D8. `POST /api/users/[userId]/api-keys` (self or admin) — `route.ts`**

Request body: `{ label?: string }`
- User can only create keys for themselves; admin can create for any user
- Generate 32-byte random key; store hash in `user_api_keys`
- Return `201 { id, keyRaw, label, createdAt }` — `keyRaw` shown once, never again

**D9. `DELETE /api/users/[userId]/api-keys/[keyId]` (self or admin) — `route.ts`**

- Set `revoked_at` on the key record; return 204

### Part E — Middleware Extension (TEAM-01)

**E1. Extend `apps/memroos/src/middleware.ts`**

Read the file first. The current middleware handles rate limiting for `/api/public/v1/*`.

Add JWT enforcement:
- For `/api/auth/*` routes: pass through (no auth required)
- For `/api/public/v1/*` routes: existing tenant API key logic (unchanged)
- For all other `/api/*` routes: extract `Authorization: Bearer` header; verify as JWT using `verifyAccessToken`; if invalid, return `401 { error: 'authentication required' }`; attach `userId` and `roles` to request headers (`x-user-id`, `x-user-roles`) for route handlers to consume
- For UI routes (`/login`, `/invite/*`): pass through
- For all other UI routes: if no valid `memroos_access` cookie (a short-lived access token stored as a non-httpOnly cookie for SSR reads), redirect to `/login`

Role enforcement for admin-only API routes (`/api/users`, `/api/auth/invite` POST):
- After JWT verification, check `x-user-roles` header; if role insufficient, return `403 { error: 'insufficient permissions' }`

**E2. Update `.env.example`**
