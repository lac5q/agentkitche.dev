# Phase 68 Plan 01: Security Boundary Hardening

## Product Goal

Close the May 2026 security review gaps by making sensitive authorization route-local, hardening prompt-injection/content scanning, and adding production-grade web security guardrails.

## Requirements Covered

- SECBOUND-01: onboarding invite authorization
- SECBOUND-02: dispatch authorization and actor derivation
- SECBOUND-03: strict capability policy defaults
- SECBOUND-04: prompt-injection ingress coverage
- SECBOUND-05: long-input scanner hardening
- SECBOUND-06: web security headers
- SECBOUND-07: auth endpoint abuse protection
- SECBOUND-08: remote-card network policy

## Implementation Plan

1. Add RED tests for route-local authorization.
   - Verify reviewer/generic API auth cannot call `/api/onboarding/invite`.
   - Verify only operator/admin identities can mint invite tokens or grant agent capabilities.
   - Verify `/api/dispatch` rejects spoofed `from_agent` and unauthenticated actor claims.

2. Implement explicit handler authorization.
   - Add onboarding invite to proxy operator routes.
   - Call the existing role/registry authorization helper inside the onboarding invite handler.
   - Add route-local dispatch authorization for operator users and authenticated agents.
   - Derive actor/from_agent from authenticated identity before audit and policy checks.

3. Tighten capability policy defaults.
   - Disable legacy undeclared capability allow mode for production and non-local profiles.
   - Preserve an explicit local-dev compatibility escape hatch if needed.
   - Ensure denied dispatch, A2A send, and memory-write attempts produce `POLICY_DENIED` audit evidence.

4. Harden prompt-injection and content scanning.
   - Route every agent-facing free-text task ingress through Iris/content scanning.
   - Cover dispatch, A2A send, hive delegation/action, and orchestration ingress where applicable.
   - Replace the long-input allow path with chunk scanning or fail-closed rejection above configured limits.
   - Add tests for payloads over 4096 characters and obvious instruction-override attempts.

5. Add web security headers and auth throttling.
   - Configure CSP, frame-ancestors or X-Frame-Options, X-Content-Type-Options, Referrer-Policy, and Permissions-Policy.
   - Add login/refresh abuse throttling or lockout telemetry separate from public API tenant rate limiting.

6. Harden A2A remote-card network policy.
   - Default private-network remote-card fetching to deny unless an explicit local-dev/private-network profile enables it.
   - Keep metadata, link-local, and loopback protections always blocked for remote cards.
   - Add tests that exercise profile-specific allow/deny behavior.

7. Update security docs and operational notes.
   - Document the route-local auth model, capability declaration policy, scanner limits, and deployment-profile network defaults.
   - Add upgrade notes for users who relied on undeclared legacy capabilities.
